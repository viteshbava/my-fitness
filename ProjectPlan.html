Fitness Training App - Project Implementation Plan

Version: 1.0
Last Updated:
Status: Ready for Implementation

1. Overview

This document outlines a step-by-step implementation plan for building the Fitness Training App. The project is divided into 6 sprints, each representing a logically independent unit of work that can be implemented, tested, reviewed, and refined before moving to the next sprint.

Implementation Approach

Each sprint follows an iterative review cycle:

  1. Implementation: Developer implements the sprint deliverables
  2. Review: User reviews the implementation and tests functionality
  3. Feedback: User provides feedback and requests changes if needed
  4. Refinement: Developer makes adjustments based on feedback
  5. Documentation Update: BRD and ProjectPlan updated if changes affect requirements
  6. Approval: User approves sprint completion before moving to next sprint

2. Sprint Overview

Sprint Name Type Description
Sprint 1 Project Foundation & Database Setup FOUNDATION Next.js setup, Supabase configuration, exercise data migration
Sprint 2 Exercise Library & Detail Views CORE Exercise browsing, search, filters, and detail display
Sprint 3 Workout Management & Calendar CORE Create/manage workouts, calendar view, exercise assignment
Sprint 4 Workout Exercise Management CORE Card interface, set management, "in progress" mode
Sprint 5 Historical Data & Progress Tracking ADVANCED Performance history, charts, max weight tracking
Sprint 6 Video Integration & Final Polish POLISH Google Drive video playback, UX improvements, deployment

3. Detailed Sprint Plans

Sprint 1: Project Foundation & Database Setup
FOUNDATION
Set up Next.js application, configure Supabase database, and migrate initial exercise data from CSV.

BRD Reference

See BRD.html Section 4 (Technical Requirements) and Section 1.1-1.2 (Exercise Database & Data Migration)

Implementation Scope

  • Initialize Next.js project with TypeScript and App Router (latest version)
  • Set up Tailwind CSS for styling
  • Configure Supabase client and environment variables
  • Create exercises table in Supabase with all required fields (per BRD Section 1.1):
    • name (text)
    • video_url (text, nullable)
    • movement_type (text)
    • pattern (text)
    • primary_body_part (text)
    • secondary_body_part (text)
    • equipment (text)
    • is_mastered (boolean, default false)
    • notes (text, nullable)
    • last_used_date (date, nullable) - for sorting
    • created_at, updated_at (timestamps)
  • Create CSV import script/utility to load exercise data into Supabase
  • Create basic home page and navigation structure
  • Move BRD and ProjectPlan to /docs folder for organization

⚠️ USER ACTIONS REQUIRED BEFORE SPRINT 1

  1. Create Supabase Account:
    • Go to supabase.com and sign up
    • Create a new project (choose a region close to you)
    • Note your database password (you'll need it)
    • Get your Project URL and anon/public key from Settings → API
  2. Prepare Exercise CSV File:
    • Ensure your CSV has columns matching the exercise database fields
    • Place the CSV file in the project root or a data folder
    • Confirm the CSV file path with the developer
  3. Create .env.local file:
    • In the project root, create a file named .env.local
    • Add the following variables: 
      NEXT_PUBLIC_SUPABASE_URL=your-project-url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

Testing & Review

  • Run npm run dev and verify app starts at localhost:3000
  • Verify Supabase connection is working (no console errors)
  • Run CSV import script and verify exercises appear in Supabase dashboard
  • Check that all exercise fields are populated correctly from CSV
  • Verify npm run build completes successfully

Success Criteria

✅ Next.js app runs locally with latest version
✅ Supabase is configured and connected
✅ exercises table created with all required fields
✅ CSV data successfully imported into Supabase
✅ Can view exercises in Supabase dashboard
✅ Project builds successfully

Sprint 3: Workout Management & Calendar
CORE
Implement workout CRUD operations, calendar view, and exercise assignment to workouts.

Dependencies

  • Sprint 1 completed (database setup)

BRD Reference

See BRD.html Section 2 (Workout Management)

Implementation Scope

  • Create workouts table in Supabase:
    • id (uuid, primary key)
    • date (date, not null)
    • created_at, updated_at (timestamps)
  • Create workout_exercises table (basic structure):
    • id (uuid, primary key)
    • workout_id (uuid, foreign key to workouts)
    • exercise_id (uuid, foreign key to exercises)
    • order_index (integer, for card ordering)
    • created_at, updated_at (timestamps)
  • Workout CRUD (Section 2.1):
    • Create new workout for any date (past or future)
    • View workout details
    • Delete workout with confirmation (show summary per Section 2.1)
    • Allow multiple workouts per day (Section 2.3)
  • Calendar View (Section 2.4):
    • Implement calendar component showing days/weeks/months
    • Display workouts on calendar days
    • Show multiple workouts per day as stacked entries
    • Allow selection of individual workouts from calendar
  • Exercise Assignment:
    • Build interface to add exercises to a workout
    • Show list of assigned exercises (basic list view, cards in Sprint 4)
    • Allow removing exercises from workout
    • Allow same exercise to be added multiple times (Section 3.2.4)

Testing & Review

  • Create workout for today's date
  • Create workout for past date and future date
  • Create multiple workouts on the same day
  • View workouts in calendar - verify they appear on correct dates
  • Select workout from calendar and view details
  • Add exercises to a workout
  • Add the same exercise multiple times to a workout
  • Remove exercises from workout
  • Delete workout with confirmation dialog

Success Criteria

✅ Can create workouts for any date (past, present, future)
✅ Can create multiple workouts on the same day
✅ Calendar displays workouts on correct dates
✅ Multiple workouts per day show as stacked entries
✅ Can select and view individual workouts
✅ Can add and remove exercises from workouts
✅ Can add same exercise multiple times
✅ Delete confirmation shows workout summary

Sprint 4: Workout Exercise Management
CORE
Build the workout exercise card interface with set management and "in progress" editing mode.

Dependencies

  • Sprint 3 completed (workouts and workout_exercises can be created)

BRD Reference

See BRD.html Section 3 (Workout Exercise Management)

Implementation Scope

  • Extend workout_exercises table with set data:
    • sets (jsonb array storing: set_number, weight, reps)
    • draft_snapshot (jsonb, for Cancel functionality)
  • Card Interface (Section 3.2):
    • Replace basic list with card-based interface (Section 3.2.1)
    • Display exercise name, completion summary, max weight
    • Show video thumbnail placeholder (implementation in Sprint 6)
    • Implement color coding: gray (no data), yellow (partial), green (complete) (Section 3.2.2)
    • Enable drag-and-drop re-ordering (order not preserved between workouts) (Section 3.2.1)
    • Implement delete with confirmation (Section 3.2.3)
  • Detail View & "In Progress" Mode (Section 3.3):
    • Create workout_exercise detail view (Section 3.3.1-3.3.2)
    • Implement read-only mode (default) and "in progress" mode
    • Block navigation when in "in progress" mode
    • Implement auto-save to database as changes are made
    • Maintain draft snapshot for Cancel functionality
    • Implement Save (clear snapshot, exit mode) and Cancel (restore snapshot, exit mode)
    • Handle browser refresh (keep auto-saved data, clear snapshot, return to read-only)
  • Sets Management (Section 3.3.3):
    • Display sets as rows with set number, weight, reps
    • Create default empty sets: 3 if new exercise, or match count from most recent workout
    • Show placeholders with previous workout data (match by set number)
    • Allow adding new sets (always at end)
    • Allow deleting last set only (minimum 1 set must remain)
    • Allow editing weight (decimal kg) and reps (integer ≥ 0)
    • Implement empty set removal on save (auto-renumber remaining)
  • Exercise Info Display (Section 3.3.2):
    • Show exercise name, max weight, notes
    • Allow editing exercise notes (same field as in Exercise Library)
    • Show video placeholder (implementation in Sprint 6)

Testing & Review

  • Card Interface:
    • Verify cards display with correct color coding (gray/yellow/green)
    • Test drag-and-drop re-ordering of cards
    • Test delete with confirmation
  • "In Progress" Mode:
    • Enter "in progress" mode and verify navigation is blocked
    • Make changes to sets and verify auto-save occurs
    • Click Save and verify snapshot is cleared and mode exits
    • Enter "in progress", make changes, click Cancel and verify data restores
    • Enter "in progress", make changes, refresh browser and verify auto-saved data persists
  • Sets Management:
    • Create new workout_exercise and verify 3 default sets (or previous count if exercise was done before)
    • Verify placeholders show previous workout data
    • Add new sets and verify they appear at the end
    • Delete last set and verify it's removed (verify cannot delete below 1 set)
    • Edit weight/reps and verify changes save
    • Leave some sets empty, save, and verify empty sets are removed

Success Criteria

✅ Workout exercises display as cards with correct color coding
✅ Cards can be re-ordered via drag-and-drop
✅ "In Progress" mode blocks navigation and auto-saves changes
✅ Save/Cancel functionality works correctly
✅ Browser refresh preserves auto-saved data
✅ Sets create with correct defaults and placeholders
✅ Can add/delete sets with proper constraints
✅ Empty sets are removed on save
✅ Exercise notes are editable

Sprint 5: Historical Data & Progress Tracking
ADVANCED
Implement historical performance tracking, progress charts, and max weight calculations.

Dependencies

  • Sprint 4 completed (workout_exercise sets can be recorded)

BRD Reference

See BRD.html Section 1.5.3-1.5.5 (Historical Performance, Max Weight, Progress Chart) and Section 3.3.4 (Historical Data View)

Implementation Scope

  • Historical Data in Workout Exercise Detail (Section 3.3.4):
    • Build historical data panel showing last 3 workouts by default
    • Display date and all sets (weight/reps) for each historical workout
    • Make panel scrollable for longer history
  • Max Weight Calculation (Section 1.5.4, 3.3.2):
    • Calculate and display single heaviest weight with ≥6 reps
    • Show max weight in workout_exercise cards (Section 3.2.1)
    • Show max weight in workout_exercise detail view (Section 3.3.2)
    • Show max weight in exercise detail view (Section 1.5.4)
  • Exercise Detail View - Historical Performance (Section 1.5.3-1.5.5):
    • Display table of all historical data grouped by session/date
    • Columns: Date, Set Number, Reps, Weight
    • Build progress chart (last 12 months):
      • X-axis: Actual dates
      • Dual Y-axes: Weight (left) and Reps (right), color-coded
      • Line chart with two lines (weight and reps)
    • Integrate charting library (Recharts or Chart.js)
  • Last Used Date Update:
    • Update exercise.last_used_date when workout_exercise is saved with data
    • Enable "Sort by Last Used" functionality from Sprint 2

Testing & Review

  • Record sets for an exercise across multiple workouts on different dates
  • Open workout_exercise detail and verify historical data shows last 3 workouts
  • Verify max weight calculation is correct (heaviest with ≥6 reps)
  • Verify max weight displays in cards and detail views
  • Navigate to exercise detail view and verify historical table shows all sessions
  • Verify progress chart displays with correct axes and data points
  • Add more historical data and verify chart updates (last 12 months only)
  • Verify last_used_date updates after recording workout data
  • Test "Sort by Last Used" in exercise library

Success Criteria

✅ Historical data panel shows last 3 workouts in workout_exercise detail
✅ Max weight calculates correctly (≥6 reps criterion)
✅ Max weight displays in all required locations
✅ Exercise detail shows complete historical performance table
✅ Progress chart displays with dual Y-axes and correct data
✅ Chart shows last 12 months of data only
✅ last_used_date updates when workout data is recorded
✅ Sort by Last Used works in exercise library
