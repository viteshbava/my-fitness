Business Requirements Document (BRD)

Fitness Training App

Date Created: January 2, 2026
Last Updated: January 6, 2026

Document Purpose

This document outlines the functional requirements for a personal fitness training application. The purpose is to establish a clear understanding of what features and capabilities the app needs before development begins.

Functional Requirements

1. Exercise Management

1.1 Exercise Database

The application will maintain a database of exercises with the following attributes:

  • Name: The exercise name
  • Video URL Link: A Google Drive link to an instructional video
  • Movement Type: Classification of the movement
  • Pattern: The movement pattern category
  • Primary Body Part: The main body part targeted by the exercise
  • Secondary Body Part: Secondary body parts engaged during the exercise
  • Equipment: Required equipment for the exercise
  • Is Mastered: Boolean flag indicating whether the exercise has been learned/mastered
  • Notes: Free-text field for additional information or instructions about the exercise

1.2 Data Migration

Initial exercise data will be imported via CSV file upload.

1.3 CRUD Operations

Phase 1 will support Read-only operations for exercises, with the exception of the Notes field which will be editable. Full CRUD (Create, Read, Update, Delete) functionality will be considered for future phases.

1.4 Exercise Library View

The application will provide a library view to browse and access all exercises in the system.

1.4.1 Search Functionality
  • Users can search exercises by exercise name
1.4.2 Filter Functionality

Users can filter the exercise library by the following criteria:

  • Movement Type
  • Pattern
  • Primary Body Part
  • Secondary Body Part
  • Equipment
  • Is Mastered (Yes/No)
1.4.3 Sorting Functionality

Users can sort exercises by:

  • Alphabetical by Name (default)
  • Last Used Date

1.5 Exercise Detail View

When a user selects an exercise from the library, a detailed view will be displayed with the following information and capabilities:

1.5.1 Exercise Attributes Display
  • Display all exercise attributes (Name, Video URL Link, Movement Type, Pattern, Primary Body Part, Secondary Body Part, Equipment, Is Mastered) in read-only mode
  • The Notes field will be editable, allowing users to update equipment settings and other relevant information
1.5.2 Video Display
  • Display a video thumbnail when a Video URL Link is present
  • When the thumbnail is clicked, display the playable video in a modal window
  • When no Video URL Link is present, display "No video available" text
1.5.3 Historical Performance Data

Display a table showing all historical workout data for the exercise, grouped by session:

  • Columns: Date, Set Number, Reps, Weight
  • Each session groups all sets performed on that date
1.5.4 Maximum Weight Score
  • Display the single heaviest weight where the user completed at least 6 reps in any set
  • This serves as the current "score" or personal record for the exercise
1.5.5 Progress Chart

Display a line chart showing progression over time:

  • X-axis: Actual dates
  • Y-axes: Two Y-axes (left and right) for Weight and Reps, color-coded
  • Data range: Last 12 months of historical data
  • Chart type: Line chart with two color-coded lines (one for weight, one for reps)

2. Workout Management

A workout represents a visit to the gym where exercises are performed. Users can manage workouts both retroactively and for future dates.

2.1 Workout CRUD Operations

Users can perform full CRUD operations on workouts:

  • Create: Create a new workout for any date (past or future)
  • Read: View workout details and assigned exercises
  • Update: Modify workout details and exercise assignments
  • Delete: Remove a workout from the system
    • Requires confirmation before deletion
    • Confirmation message displays summary including: date, number of exercises, and any recorded data

2.2 Workout Attributes

Each workout has the following attributes:

  • Date: The date of the workout
  • Exercises: A list of exercises assigned to this workout (workout_exercises)

Note: Start time, end time, duration, and workout status are not captured in Phase 1.

2.3 Multiple Workouts Per Day

Users can create multiple workouts on the same day (e.g., morning and evening sessions).

2.4 Calendar View

Workouts will be displayed in a calendar interface:

  • Granularity: Individual days grouped into weeks and months
  • Multiple workouts per day: Displayed as stacked entries within the day cell
  • Selection: Each individual workout can be selected from the calendar
  • Preview information: No preview information displayed on calendar (Phase 1)

3. Workout Exercise Management

A workout_exercise represents a specific exercise assigned to a workout, along with its associated sets, reps, and weight data.

3.1 Workout Exercise CRUD Operations

Users can perform full CRUD operations on workout_exercises:

  • Create: Add an exercise to a workout (with default empty sets)
  • Read: View exercise details, sets, and historical data
  • Update: Modify sets, weight, reps, and other details
  • Delete: Remove an exercise from a workout (with confirmation)

3.2 Workout Exercise List View (Card Interface)

When viewing a workout, exercises are displayed as a stacked list of cards:

3.2.1 Card Display
  • Layout: Vertical stack of cards, one per workout_exercise
  • Summary Text: Each card displays exercise name and completion summary (e.g., "Bench Press - 4 sets completed")
  • Historical Max Weight: Display the maximum weight (6+ reps) for this exercise
  • Video Thumbnail: If the exercise has a video URL, display a small thumbnail on the left side of the card. If no video exists, no thumbnail is shown
  • Re-ordering: Cards can be re-ordered by dragging (order is not preserved between workouts)
3.2.2 Visual Indicators (Color Coding)

Cards use color to indicate data completion status:

  • No data: Gray/neutral color (no sets have weight/reps recorded)
  • Partial data: Yellow/amber color (some sets filled, some empty)
  • All sets filled: Green color (all sets have weight/reps recorded)
3.2.3 Card Interactions
  • Click to open: Clicking a card opens the workout_exercise detail view
  • Delete with confirmation: Option to delete a workout_exercise
    • Confirmation message required before deletion
    • If data is recorded, display summary of recorded data in confirmation message
3.2.4 Duplicate Exercises

The same exercise can be added multiple times to a single workout (e.g., warm-up sets and working sets as separate entries).

3.3 Workout Exercise Detail View

When a workout_exercise card is opened, users can view and edit exercise details, sets, and historical data.

3.3.1 "In Progress" Mode

The detail view has two modes: read-only and "in progress" (editable):

Read-Only Mode (Default):

  • All data is displayed but not editable
  • User can enter "in progress" mode to make changes
  • Navigation to other views is allowed

"In Progress" Mode (Editable):

  • All fields become editable
  • Navigation is blocked - user cannot leave this view
  • Changes are auto-saved to database continuously as they are made
  • A snapshot of the initial state is maintained for Cancel functionality
  • Two exit options:
    • Save: Clears draft snapshot, exits "in progress" mode, returns to read-only view
    • Cancel: Shows "are you sure" prompt, restores data from snapshot if confirmed, exits "in progress" mode
  • If user refreshes browser while in progress:
    • Auto-saved changes remain in database
    • Status automatically returns to read-only mode
    • Draft snapshot is cleared

Multi-Device Behavior:

  • Application is designed for single-user, single-device use
  • If accessed from multiple devices (rare scenario): last-write-wins approach
  • No conflict detection or device locking in Phase 1
3.3.2 Exercise Information Display
  • Exercise Name: Display the exercise name
  • Video Display: Display below the exercise name but above the sets list
    • If the exercise has a video URL, display a video thumbnail
    • When the thumbnail is clicked, display the playable video in a modal window (same functionality as in Exercise Library)
    • If no video URL exists, display "No video available" text
  • Historical Max Weight: Display the maximum weight (6+ reps) achieved for this exercise across all workouts
  • Exercise Notes: Display and allow editing of global exercise notes (same notes field as in Exercise Management)
3.3.3 Sets Management

Set Display:

  • Sets are displayed as a list of rows
  • Each row shows: Set number, Weight (kg), Reps
  • Weight and reps are editable when in "in progress" mode

Default Empty Sets:

  • When creating a workout_exercise, empty sets are added automatically:
    • If this exercise was done in a previous workout: default to the number of completed sets from the most recent workout where this exercise was done (only count sets with weight/reps recorded, ignore empty sets)
    • If no previous workout exists: default to 3 empty sets
    • This logic applies to each instance if the same exercise is added multiple times to a workout

Set Placeholders:

  • For each set, display placeholder text showing the weight/reps from the most recent workout where this exercise was done:
    • Match by set number (set 1 shows previous set 1, set 2 shows previous set 2, etc.)
    • If current set number exceeds previous workout's set count, show blank placeholder
    • Example: Previous workout had 5 sets, current workout has 6 sets - sets 1-5 show previous data, set 6 has blank placeholder

Adding Sets:

  • User can add new empty sets
  • New sets are always added at the end of the list

Deleting Sets:

  • User can only delete the last set in the list
  • Deletion happens immediately without confirmation
  • At least one set must always remain (cannot delete the first/only set)

Editing Sets:

  • User can edit weight and reps for any set in the list (when in "in progress" mode)
  • Weight: numeric value in kg, can be decimal (e.g., 22.5 kg)
  • Reps: positive integer, minimum 0
  • If weight is entered but reps is left empty, assume reps = 0
  • If reps is entered but weight is left empty, assume weight = 0
  • A set is considered "completed" (has data) if either weight OR reps has a value (including 0)
  • A set is only considered "empty" if both weight AND reps are not entered

Empty Set Removal:

  • When saving (both manual Save and auto-save), automatically remove all empty sets
  • After removal, renumber remaining sets sequentially
  • Example: Set 1 (completed), Set 2 (empty), Set 3 (completed), Set 4 (empty) â†’ After save: Set 1 (completed), Set 2 (completed - previously Set 3)
3.3.4 Historical Data View

A separate section/panel within the workout_exercise detail view displays historical performance data:

  • Display format: Scrollable list of previous workouts
  • Data shown: Each historical workout entry shows:
    • Date of the workout
    • All sets from that workout displayed as rows with weight and reps
  • Number of sessions: Display last 3 workouts by default
  • Accessibility: Available while working out (within the workout_exercise detail view)

4. Technical Requirements

4.1 Frontend Framework

The application will be built using the latest version of Next.js as the primary frontend framework.

4.2 Database

The application will use Supabase as the database solution for data storage and retrieval.

4.3 Mobile-First Design

The application will be primarily used on mobile devices at the gym. The user interface must be optimized for mobile screens first, with desktop support as a secondary consideration.

Design Priorities:

  • Mobile screens are the primary target device
  • All features must be fully functional and easily accessible on mobile
  • Touch-friendly interface elements (buttons, cards, form inputs)
  • Responsive design that adapts to desktop for occasional use
  • Performance optimized for mobile networks

4.4 Deployment & Testing

The application will be deployed to Vercel from the start of development to enable real-world mobile testing throughout the build process.

Deployment Requirements:

  • Deploy to Vercel after Sprint 1 (Project Foundation) is complete
  • Continuous deployment from the main Git branch
  • Environment variables configured in Vercel for Supabase connection
  • Ability to test on actual mobile devices throughout development
  • Vercel preview deployments for testing changes before merging
